# Work Page Refinement Execution Plan

## 目标

在保留现有三栏结构的前提下，把 Work Page 从“结构成立”推进到“可交付交互成品”。

本轮只解决 3 个核心问题：

1. 用户一眼看清 **当前状态**
2. 用户一眼看清 **现在最推荐的动作**
3. 用户一眼看清 **当前动作对应右侧计划的哪一步**

---

## 本轮范围

### 必做

- 重构顶部状态区
- 收紧中间“当前需要你决定”主卡
- 强化右侧任务计划的当前步骤与操作入口
- 保持中间主区 / 左栏 / 右栏语义分工不变

### 本轮不做

- 整页重新布局
- 复杂动画和视觉风格重设计
- 真正的 AI 计划生成升级
- 左栏的大规模重构
- Work 之外页面联动改造

---

## 设计原则

1. **不重做框架，只收敛主流程**
2. **状态、阶段、异常必须拆开表达**
3. **主卡只保留决策必需信息**
4. **右侧计划不只是展示，要服务当前动作**
5. **中间主区优先于左侧背景信息**

---

## 交付结果

完成后，Work Page 应该形成下面的阅读顺序：

### 顶部

- 任务标题
- 任务状态
- 当前阶段
- 当前异常

### 左侧

- 任务背景
- 日程信息
- 当前阻塞

### 中间

- 当前需要你决定
- 任务进展
- 最新结果 / 空结果
- 输入区

### 右侧

- 任务计划
- 当前步骤高亮
- 恢复 / 重规划入口

---

## Phase 1：重构顶部状态区

### 目标

解决当前“任务生命周期 / 当前协作阶段 / step chips”重复表达的问题。

### 当前问题

顶部现在混合了：

- 生命周期
- 阶段
- 中断/失败信息

用户会分不清：

- 当前到底是进行中、失败、还是已中断
- `进行中` 和 `执行中` 的关系
- “异常”到底是状态还是阶段

### 改造要求

顶部改成 3 层表达：

#### 1. 任务状态

只允许表达任务当前总体状态，例如：

- 进行中
- 已中断
- 已完成
- 待开始

#### 2. 当前阶段

只允许表达当前协作阶段，例如：

- 理解任务
- 执行中
- 等待确认

#### 3. 当前异常

只允许表达异常/阻塞，例如：

- 网络错误，等待恢复
- 等待审批
- 等待补充说明

### 实施要求

- 不再并列展示“生命周期 chips + 协作阶段 chips + 当前摘要”三套相近信息
- “执行中断”不再作为阶段文案出现
- 当前异常为空时，不强行占位大块区域

### 主要文件

- `src/components/work/work-page-client.tsx`

### 验收标准

- 用户 3 秒内知道任务是否正常推进
- 用户能明确区分状态、阶段、异常
- 顶部信息比现在更短、更直接

---

## Phase 2：收紧中间“当前需要你决定”主卡

### 目标

让主卡从“状态说明卡”变成真正的“决策卡”。

### 当前问题

主卡里目前混合了：

- 状态说明
- 建议
- 操作
- 次操作
- 提醒
- 背景证据

内容都相关，但动作判断不够集中。

### 目标结构

主卡必须收敛成固定结构：

#### 当前状态

例如：

- 上一轮执行已中断
- Agent 正等待你的审批
- 最新结果已经生成

#### 建议动作

例如：

- 先恢复执行；如果你想调整方向，再补充说明后执行
- 先完成审批决定，再继续本次执行
- 先确认结果，再决定继续还是收尾

#### 主操作

只保留 1~2 个最推荐动作，例如：

- 恢复执行
- 补充说明后执行
- 处理审批
- 确认结果

#### 次操作

只保留低优先级操作，例如：

- 调整日程
- 暂不处理

#### 补充提醒

只保留真正影响决策的一条摘要，例如：

- 最近失败原因：网络错误 / 非 101 状态码
- 当前阻塞：等待审批

### 实施要求

- 把系统字段、里程碑、重复背景从主卡移出去
- 主卡首屏不出现大段“相关背景列表”
- 主卡视觉上必须是页面一级强调区
- 主按钮与次按钮的层级必须拉开

### 主要文件

- `src/components/work/work-page-client.tsx`

### 验收标准

- 用户打开页面后，不需要读完整张卡也知道下一步该点什么
- 主卡只回答一个问题：**你现在最该做什么**

---

## Phase 3：强化右侧任务计划为“全局导航”

### 目标

让右侧计划从“可看”升级为“可导航、可支撑当前动作”。

### 当前问题

右侧现在已经有计划结构，但更像静态说明：

- 有步骤
- 有状态
- 有说明

但不够明确：

- 现在卡在哪一步
- 当前步骤是否阻塞
- 当前动作是否与这一步对应

### 改造要求

计划区必须明确展示：

#### 1. 计划整体状态

例如：

- 已根据当前上下文更新 · 4 步
- 当前为占位计划
- 最近更新时间

#### 2. 当前步骤高亮

当前步骤必须同时表达：

- 步骤名
- 是否为当前步骤
- 当前状态：进行中 / 已中断 / 等待你确认 / 已完成 / 阻塞

#### 3. 当前步骤操作

当存在当前步骤时，直接提供操作入口，例如：

- 从这一步恢复
- 重新规划后继续

#### 4. 其它步骤降级

非当前步骤只保留：

- 标题
- 状态
- 简短目标

### 文案要求

步骤命名统一成可执行风格：

- 梳理目标与约束
- 补齐上下文
- 推进主产出
- 确认结果与下一步

不要混用过多不同语气的动词。

### 实施要求

- 当前步骤必须在视觉上明显强于其它步骤
- 计划状态、步骤状态、当前动作要形成对应关系
- 如果当前步骤阻塞，右栏一眼可见阻塞状态

### 主要文件

- `src/components/work/task-plan-side-panel.tsx`
- `src/modules/queries/get-work-page.ts`
- `src/components/work/work-page-client.tsx`

### 验收标准

- 用户看右栏 3 秒内知道当前卡在哪一步
- 用户能直接从右栏理解当前该恢复还是该重规划

---

## Phase 4：中间“任务进展”区收敛

### 目标

让中间流不再像系统记录，而像真正的协作主线。

### 改造要求

把中间流明确分成 3 类卡：

#### A. 过程记录卡

例如：

- 任务有新的进展
- 执行过程中出现阻塞

这类卡是背景过程，视觉权重要低。

#### B. 决策节点卡

例如：

- 等待你的审批
- 需要补充说明
- 当前建议恢复执行

这类卡是当前主线，视觉权重要高。

#### C. 结果卡 / 空结果卡

例如：

- 已生成结果
- 尚未生成结果

这类卡必须与普通过程卡明显不同。

### 命名要求

把“对话 / 协作流”优先改名为：

- `任务进展`

如果后续更偏人与 Agent 的交流，再考虑恢复更偏“协作”的命名。

### 主要文件

- `src/components/work/collaboration-stream.tsx`
- `src/components/work/work-page-client.tsx`

### 验收标准

- 用户能区分背景记录、当前决策节点、结果反馈
- 决策节点卡明显强于普通过程卡

---

## Phase 5：左栏压缩（次优先级）

### 目标

让左栏成为辅助判断区，而不是与中间主区争重心。

### 改造要求

左栏优先保留：

- 任务背景
- 日程信息
- 当前阻塞

其它信息降级或合并：

- 当前执行
- 最近运行时间
- 最近失败原因

### 推荐分组

#### 基本信息

- 优先级
- 截止时间
- 日程窗口

#### 运行信息

- 当前阻塞
- 最近失败原因
- 最近运行状态

### 主要文件

- `src/components/work/run-side-panel.tsx`

### 验收标准

- 左栏明显缩短
- 用户一眼先看到阻塞/失败原因，而不是基础字段

---

## 实施顺序

严格按下面顺序执行：

1. **Phase 1：顶部状态区**
2. **Phase 2：中间主卡**
3. **Phase 3：右侧任务计划**
4. **Phase 4：任务进展区**
5. **Phase 5：左栏压缩**

原因：

- 先收敛状态表达
- 再收敛主动作
- 再建立与计划步骤的对应关系
- 最后处理信息密度与辅助区

## 实施检查清单

每完成一阶段，都检查这 6 条：

1. 用户是否能在 3 秒内看懂当前状态
2. 用户是否能在 5 秒内找到推荐动作
3. 用户是否知道当前动作对应右侧哪一步
4. 左栏是否仍然只是辅助信息
5. 中间主卡是否仍然是一级重心
6. 是否减少了重复表达，而不是新增一套表述
